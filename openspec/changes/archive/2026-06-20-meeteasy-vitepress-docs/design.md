## Context

wiki 仓库目前仅有 OpenSpec 工作流配置，尚无文档站点。会易小蜜书（MeetEasy）产品文档分散在 meeteasy 主仓库（`README.md`、`docs/architecture_design.md`、`openspec/project.md`、VisuSpace DSL 规范、插件指南等），缺乏统一入口与对外展示能力。

本变更在 wiki 仓库搭建 VitePress 静态文档站，归纳**会易小蜜书（MeetEasy）**的产品能力、用户操作路径与技术架构，面向三类读者：潜在客户/决策者（产品介绍）、会务组织者与参会者（使用手册）、开发者与运维（技术文档）。

## Goals / Non-Goals

**Goals:**

- 在 wiki 仓库建立可本地开发与构建的 VitePress 站点
- 支持 `zh-CN` / `en-US` 双语切换
- 支持深色与浅色主题切换（跟随系统或手动切换）
- 覆盖产品介绍、使用手册、技术介绍、法律法规、关于我们五大模块，信息架构清晰
- 缺失截图使用统一占位图，并标注 `<!-- TODO: 补充截图 -->` 供人工后续替换
- 站点品牌 Logo 与主题色与 MeetEasy 产品 UI（MeetApp / Console）对齐
- 产品品牌命名：中文「会易小蜜书」、英文「MeetEasy」；公司主体：星汇盛世（北京）科技有限公司
- 法律法规、关于我们文案按业界最佳实践撰写完整结构与正文
- 提供 `dev` / `build` / `preview` 脚本与简要 README

**Non-Goals:**

- 不修改 meeteasy 主应用代码或 API
- 不实现 API 文档自动生成（OpenAPI/Swagger 集成留作后续迭代）
- 不实现全文搜索服务（先用 VitePress 内置本地搜索）
- 不部署到生产环境（仅提供构建产物与部署说明）
- 不翻译 meeteasy 源码注释或完整 DSL_SPEC（技术文档做摘要与链接引用）

## Decisions

### 0. 产品品牌命名

| 语言 | 品牌名称 | 使用场景 |
|------|----------|----------|
| 中文 | **会易小蜜书** | 中文站点标题、Hero、导航、关于我们、法律法规正文 |
| 英文 | **MeetEasy** | 英文站点标题、Hero、导航、About、Legal 正文 |

**命名规则**：

- 中文 locale 首选「会易小蜜书」，首次出现可附注英文 MeetEasy，如「会易小蜜书（MeetEasy）」
- 英文 locale 首选「MeetEasy」，About 页可注明中文品牌名「会易小蜜书」
- 法律文档中平台名称统一写作「会易小蜜书（MeetEasy）平台」/ "MeetEasy (会易小蜜书) Platform"
- 代码、仓库、技术文档中的 `meeteasy` 包名/API 路径保持不变，不作翻译

**公司主体**：

| 字段 | 中文 | 英文 |
|------|------|------|
| 公司全称 | **星汇盛世（北京）科技有限公司** | Xinghui Shengshi (Beijing) Technology Co., Ltd. |

法律文档、关于我们、隐私政策中的「数据控制者」「运营主体」「本公司」均指上述公司。其余事实性字段（统一社会信用代码、注册地址、联系邮箱、管辖法院等）仍使用 `[待补充]`，待人工补充。

### 1. 站点目录结构：`src/` 作为 VitePress 根目录

**选择**：使用 wiki 仓库已有的 `src/` 目录作为 VitePress 站点根，`src/.vitepress/` 存放配置；仓库根目录保留 `package.json` 与 `openspec/` 等工作流文件。

**理由**：与 wiki 仓库目录约定一致（源码集中在 `src/`），根目录保持简洁，便于后续扩展其他 `src` 子项目。

**备选**：根目录 `docs/` — 与 wiki 既有 `src/` 布局不一致，且易与 meeteasy 仓库 `docs/` 混淆。

### 2. 国际化：VitePress 内置 `locales` 配置

**选择**：使用 VitePress `themeConfig.locales` 实现 `root`（中文）与 `en`（英文）双语，各语言维护独立 Markdown 文件（`src/zh/` 与 `src/en/`）。

**推荐结构**：

```
wiki/
├── package.json           # 构建脚本（vitepress dev/build 指向 src）
├── README.md
├── openspec/
└── src/                   # VitePress 站点根目录
    ├── .vitepress/
    │   ├── config.ts      # 主配置（locales、nav、sidebar）
    │   └── theme/         # 自定义主题扩展（如需要）
    ├── zh/
    │   ├── index.md
    │   ├── product/
    │   ├── user-manual/
    │   ├── developer/
    │   ├── legal/           # 法律法规
    │   └── about/           # 关于我们
    ├── en/
    │   └── （镜像结构）
    └── public/
        └── images/
            ├── logo.png         # MeetEasy 品牌 Logo（蜜蜂吉祥物）
            └── placeholder.svg
```

**理由**：VitePress 官方 i18n 方案成熟，SEO 友好，支持语言切换器。

**备选**：单文件双语（frontmatter `lang`）——维护成本高，不利于侧边栏与导航分离。

### 5. 法律法规与关于我们文案规范（业界最佳实践）

**选择**：法律法规与关于我们页面按 SaaS / 会务平台业界最佳实践撰写**完整章节结构与正文**，结合 MeetEasy 实际功能定制；公司全称已确定为**星汇盛世（北京）科技有限公司**；统一社会信用代码、注册地址、客服邮箱等其余事实性字段使用 `[待补充]` 占位，页首标注「发布前需法务/品牌审核」。

**理由**：空占位正文无法满足合规展示与品牌信任需求；按最佳实践模板撰写可大幅减少后续法务返工，同时保留人工核验入口。

#### 5.1 法律法规 — 通用写作规范

每篇法律文档须包含：

- **元信息**：生效日期、最后更新日期、版本号
- **目录**：章节锚点导航（VitePress 侧边栏 + 文内 TOC）
- **可读性**：短段落、编号条款、关键术语首次出现时释义
- **双语一致**：中英文法律文档信息架构对齐，英文参考 GDPR / SaaS ToS 惯例

#### 5.2 隐私政策（`/legal/privacy`）必备章节

参考《个人信息保护法》（PIPL）及 GDPR 透明性原则，须覆盖：

| 章节 | 内容要点（结合 MeetEasy） |
|------|--------------------------|
| 引言与适用范围 | 适用产品（Admin / Console / MeetApp / 微信小程序）、适用人群 |
| 数据控制者信息 | 星汇盛世（北京）科技有限公司、注册地址、隐私联系人 `[待补充]` |
| 收集的个人信息类型 | 账号（手机/邮箱/微信 OpenID）、报名信息、设备标识、日志与分析数据、AI 对话内容、音视频（ASR） |
| 收集目的与法律依据 | 账户管理、会务服务履约、安全风控、产品改进、营销（需单独同意） |
| Cookie 与本地存储 | 会话 Token、语言/主题偏好、分析 Cookie 说明 |
| 第三方共享与委托处理 | 微信、LLM/ASR 服务商、云基础设施；列出类别而非虚假具体名称 |
| 跨境传输 | 如有境外模型服务，说明传输场景与保障措施 |
| 存储期限 | 按数据类型分级保留策略 |
| 用户权利 | 查阅、复制、更正、删除、撤回同意、注销账号；行使方式与响应时限 |
| 未成年人保护 | 不满 14 周岁需监护人同意 |
| 安全措施摘要 | 加密、访问控制、多租户隔离（引用数据安全页） |
| 政策更新 | 修订通知方式 |
| 联系我们 | 隐私投诉/咨询渠道 `[待补充]` |

#### 5.3 服务条款（`/legal/terms`）必备章节

参考主流 SaaS ToS（Stripe / Notion / 国内云服务条款结构）：

- 服务定义与范围、服务变更与中断
- 账户注册资格、账号安全责任
- 付费服务与退款政策（如有）
- 用户内容与知识产权（会务数据归属组织者）
- 禁止行为（违法内容、爬虫、干扰服务、滥用 AI）
- 免责声明与责任限制
- 赔偿条款、终止与暂停
- 争议解决与适用法律（中国大陆法律 + 管辖法院 `[待补充]`）
- 条款修改机制

#### 5.4 用户协议（`/legal/user-agreement`）必备章节

面向终端用户（参会者、组织者），侧重账号与使用行为规范：

- 协议接受方式（注册/登录/使用即视为同意）
- 账号注册与实名要求
- 会务报名、表单提交、AI 互动的用户义务
- 内容合规（不得发布违法、侵权、骚扰信息）
- AI 生成内容免责声明（仅供参考，不构成专业建议）
- 账号冻结、注销与数据删除
- 协议修订与用户通知

#### 5.5 数据安全（`/legal/data-security`）必备章节

参考 ISO 27001 / SOC 2 公开摘要页惯例：

- 安全治理与组织措施
- 技术措施：传输加密（HTTPS/TLS）、存储保护、备份与恢复
- 访问控制：RBAC、多租户隔离、最小权限
- 日志审计与入侵检测
- 漏洞管理与安全更新
- 事件响应流程（发现 → 遏制 → 通知 → 复盘）
- 合规承诺：PIPL、网络安全法、等级保护（如适用，标注 `[待确认]`）
- 子处理者管理原则
- 安全报告渠道 `[待补充]`

#### 5.6 关于我们 — 通用写作规范

参考 SaaS 官网 About 页最佳实践（清晰、可信、行动导向）：

- **叙事结构**：问题 → 解决方案 → 价值证明，避免空洞口号
- **受众分层**：分别点明服务组织者、参会者、平台运营方
- **信任信号**：产品能力描述（多租户、全流程会务管理），量化数据 `[待补充]`
- **行动号召（CTA）**：引导至产品介绍、联系我们、使用手册

#### 5.7 关于我们各页内容要点

| 页面 | 最佳实践要点 |
|------|-------------|
| 公司介绍 `/about/` | 星汇盛世（北京）科技有限公司与会易小蜜书产品介绍、核心能力摘要、服务对象、发展历程 `[待补充]` |
| 使命愿景 `/about/mission` | 使命（为何做会务数字化）、愿景（3–5 年目标）、核心价值观 3–5 条（如「租户优先」「开放扩展」「体验至上」），与产品原则呼应 |
| 团队介绍 `/about/team` | 团队规模与文化 `[待补充]`、组织架构概览（产品/研发/运营）、招聘 CTA `[待补充]`；头像可用占位图 |
| 联系我们 `/about/contact` | 分渠道展示：商务合作、技术支持、媒体问询、隐私合规；每项含邮箱 `[待补充]`、预期响应时间、工作时间 |

### 6. 图片占位策略

**选择**：`src/public/images/placeholder.svg` 作为通用占位图；Markdown 中使用 `![描述](/images/placeholder.svg)` 并在 alt 或相邻注释标明待补充内容。

**理由**：构建不报错，人工替换时只需改路径，占位图风格统一。法律法规与关于我们的**正文**不适用空占位策略，仅截图与事实性字段可占位。

### 7. 内容来源与同步策略

**选择**：首版内容手写归纳自 meeteasy 仓库现有文档，不建立自动同步管道。

**理由**：wiki 与 meeteasy 分仓，自动同步需额外 CI；首版以「可用」为目标，后续可加脚本从 meeteasy 拉取 README 片段。

### 8. 包管理与脚本

**选择**：wiki 根目录 `package.json`，依赖 `vitepress`（最新稳定版），通过 `src` 参数指定站点根目录，脚本：

- `docs:dev` — `vitepress dev src`，本地开发
- `docs:build` — `vitepress build src`，生产构建
- `docs:preview` — `vitepress preview src`，预览构建产物

### 9. 品牌 Logo 与 Favicon

**选择**：从 meeteasy 仓库复制品牌 Logo 至 wiki 站点，配置导航栏 Logo 与 Favicon。

| 资源 | 来源路径 | 目标路径 | 说明 |
|------|----------|----------|------|
| Logo | `meeteasy/src/frontend/visuspace/src/assets/logo.png` | `src/public/images/logo.png` | 蜜蜂吉祥物，与产品一致 |
| Favicon | 同上（或裁切） | `src/public/favicon.png` | 浏览器标签图标 |

**VitePress 配置**：

```ts
// themeConfig — 按 locale 分别配置
// zh: siteTitle: '会易小蜜书'
// en: siteTitle: 'MeetEasy'
logo: '/images/logo.png',
```

首页 Hero 区域可展示 Logo 图片；导航栏左侧显示 Logo + 站点名称。

**理由**：Logo 是品牌识别的核心元素，文档站须使用与产品相同的视觉标识。

## Risks / Trade-offs

| 风险 | 缓解 |
|------|------|
| 文档与 meeteasy 代码不同步 | 在开发者文档注明内容来源与最后更新日期；后续可加 CI 检查 |
| 双语维护工作量翻倍 | 首版核心页面双语，次要页面可先中文后补英文 |
| 截图缺失影响可读性 | 统一占位图 + TODO 注释，列出待补充截图清单 |
| 法律文案与产品功能不一致 | 基于 MeetEasy 实际功能清单撰写，法务审核前标注草稿状态 |
| VitePress 版本升级 breaking change | 锁定 `package.json` 版本范围，升级时跑 build 验证 |

## Migration Plan

1. 在 `src/` 初始化 VitePress 项目与配置
2. 创建目录骨架与中英文首页
3. 按模块填充内容（产品 → 使用手册 → 技术 → 法律法规 → 关于我们）
4. 复制品牌 Logo、配置 `custom.css` 品牌色与 Favicon
5. 添加占位图与 README 说明
6. 本地 `docs:build` 验证无 broken links
7. （可选）配置 GitHub Pages / Nginx 静态托管

无回滚风险——纯新增，不影响现有 wiki 内容。
